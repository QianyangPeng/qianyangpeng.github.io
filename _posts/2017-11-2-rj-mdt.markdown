~Using Disjunctive Normal Form to encoded rules based Decision Tree~

As a computer scientist, not only can you work in technological companies, but also you can work with people from all trades and professions. However, some people are not really good at using computers, such as elder medical practitioners. That is, you need to resolve the dilemma of a high accessibility and a complicated functionality. From the front-end, you can implement a powerful interface, with a user-friendly interaction design. While from the back-end, you can achieve this by having a delicate data structure and architecture design, to make your system robust, extensible and flexible.

I faced with this challenge when I had an internship in Shanghai Ruijin Hospital. It is an open source solution so we can discuss it here together. The department I work in is the breast cancer department.(The content below is containing words about cancer and some of them may be discomforting.)

In Oncology, we have clinical practice guidelines to help us decide the *rehabilitation therapy* for patients after surgery. Typically there are 4 types of therapies:

- Radiation therapy
- Chemotherapy
- Endocrine therapy
- Targeted therapy

In each of the therapy, there are many kinds of sub-therapy to choose from. For example, if the chosen therapy is chemotherapy, we need to go a step further and choose a specific therapy from:

- P
- Tcb
- Ec-T(wp)
- ddAC-ddP
- etc

There are many factors that can influence the decision of choosing a therapy. A classical example is "a radiation therapy is not suitable for patients older than 70". If we have a complete medical record of a patient, we can follow a guideline to make a decision, that which rehabilitation therapy we should to choose to apply on him/her. A famous guideline is called *NCCN Clinical Practice Guidelines in Oncology*, in which NCCN stands for *National Comprehensive Cancer Network*. The decision tree of choosing a specific chemotherapy is:

![Nccn](https://qianyangpeng.github.io/statics/MDT/nccnchomo.png)

So in terms of chemotherapy, the main laboratory indicators we need to know is *HR*, *HER-2*, *ALN* and the size of foci.

Before computer science technology is introduced into this hospital. This decision process is basically based on doctor's experience, with the reference of such a decision table. Doctors never fully depend on this table because the decision model is not linear. For example, no chemotherapy should be conduct on people older than 70, but how about a 69 year-old patient? No chemotherapy should be conduct on people whose foci size not bigger than 0.5cm, but how about a patient whose foci is 0.501cm? So actually this is a complicated problem and really in need of experienced experts to make decision based on their rich experience.

However, with the introduction of computer science related technology, this process could be fit into some machine learning models such as neural networks. After Training the data with large data-set of past medical record of our hospital, we can get a model with can infer the decision result with the maximum likelihood. We are not going to let machines to completely play the role of expert, but the decision result of machine is a good subsidiary to assist the discussing process of experts meetings.

To build all these things from scratch, the first step is to implement a basic module to directly give the infer result based on given decision trees without considering any non-linear feature of the model. This is the basis of all the machine learning model which is all based on these decision trees. And this is exactly my part of work of my internship. I need to encode two set of decision trees, each set with 4 big categories of therapies.

Thus the targeted module need to have the features below:

- Accurate representation of the decision tree
- Straightforward visualization of the decision tree
- Let doctors modify the decision tree easily, without needing to directly modify the code and data file
- High Maintainability after software delivery

To design this decision tree parsing module, I used two ideas to achieve the four features above:

- Separate rule file and code file
- Using Disjunctive Normal Form to encode the rules in the rule file

A decision tree can be implemented with a lot of nested if-then-else sentences, but a wiser solution is to use the code part only play the role of a driver. That is, no rule is written into code, and the work of code is only to read and parse the rule file, thus dynamically building a decision tree. Separating rule file and code file is having many advantages:

- Users can choose which rule file to use. A set of rule files can be maintained simultaneously. That is, we can keep track of different versions of rule file and the different rule of different guidelines.
- It gets easier to design the corresponding modules. The rule file can be used in visualization module, decision module and editor module. It is a good example of "high cohesion and low coupling".
- The code gets much shorter, and the debugging process gets much easier.

The second idea is to use Disjunctive Normal Form to encode the rules in the rule file. As is mentioned above, we have several modules, including visualization, decision making and editor module, that need to read and edit the rule file. Also, every node of the decision tree could have a complicated structure. In my design, a decision tre file is a JSON file, a decision tree is a dictionary, and a node in the decision tree is a tuple. The structure of a tuple is:

Index | Content | Type                                |example|
:-----|:--------|:------------------------------------|:-|
0     | NodeID  | int                                 | 2 |
1     | Rule    | list of (list of (3tuple of string))| [[("Age",">","30"),("ALN","==","False")],["HR","==","True"]] |
2     | IsLeaf  | bool                                | False |
3     | ChildID | int                                 | 16 |
4     | LeafStr | list of string                      | [] |

The *Rule* part is a "list of (list of (3tuple of string))", which should be parsed in Disjunctive Normal Form.
